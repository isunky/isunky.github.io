# isunky.github.io 使用手册

这份手册用于日常维护 `isunky.github.io` 个人站点。站点主要发布三类内容：个人日志、产品日志、产品介绍。

站点地址：

```text
https://isunky.github.io/
```

在线写作后台：

```text
https://isunky.github.io/admin/
```

注意：在线后台目前还不能完整使用，因为 GitHub OAuth 代理服务尚未配置完成。当前写作主流程仍然是原来的方式：本地编辑 Markdown 文件，提交到 GitHub，再由 GitHub Actions 自动发布。

## 1. 内容结构

站点内容都放在 `src/content` 目录下。

| 内容类型 | 目录 | 用途 | 前台页面 |
| --- | --- | --- | --- |
| 个人日志 | `src/content/logs` | 写个人思考、阶段复盘、方法总结 | `/logs/` |
| 产品日志 | `src/content/product-logs` | 记录产品进展、版本变化、关键取舍 | `/product-logs/` |
| 产品介绍 | `src/content/products` | 维护产品名称、简介、状态、仓库链接 | `/products/` |

首页会优先展示产品和产品动态。产品动态来自 `product-logs`，不要在首页代码里手写动态内容。

## 2. 当前推荐写作方式

当前优先使用原来的本地 Markdown 写作方式：

1. 在本地打开仓库 `D:\AICode\isunky`。
2. 按内容类型编辑 `src/content` 下的 Markdown 文件。
3. 本地预览确认内容和页面显示。
4. 运行检查命令。
5. 使用 Git 提交并推送到 `master`。
6. GitHub Actions 自动构建并发布站点。

适合使用本地写作的情况：

- 当前在线后台还不能登录发布。
- 需要批量修改多篇文章。
- 需要同时改页面结构、样式或站点功能。
- 希望在发布前完整跑一遍检查和构建。

在线后台地址已经部署，但暂时只作为后续入口保留：

```text
https://isunky.github.io/admin/
```

## 3. 原来的本地写作发布流程

这是目前实际可用的写作流程。

1. 进入本地仓库：

```powershell
cd D:\AICode\isunky
```

2. 启动本地开发服务器：

```powershell
npm run dev
```

3. 打开本地预览：

```text
http://localhost:4321/
```

4. 根据内容类型编辑 Markdown 文件：

| 要写的内容 | 编辑目录 |
| --- | --- |
| 个人日志 | `src/content/logs` |
| 产品日志 | `src/content/product-logs` |
| 产品介绍 | `src/content/products` |

5. 保存文件后，在浏览器里确认页面显示。

6. 发布前运行检查：

```powershell
npm test
npm run check
npm run build
```

7. 提交并推送：

```powershell
git status
git add .
git commit -m "更新站点内容"
git push origin master
```

8. 等待 GitHub Actions 发布完成。

查看部署状态：

```powershell
gh run list --repo isunky/isunky.github.io --branch master --limit 5
```

重点看名为 `Deploy GitHub Pages` 的 workflow。它成功后，线上站点就发布成功。

## 4. 在线后台发布流程

在线后台待 OAuth 配置完成后再使用。配置完成后，流程如下：

1. 访问 `https://isunky.github.io/admin/`。
2. 使用 GitHub 登录。
3. 选择要编辑的集合：
   - `Personal Logs`：个人日志
   - `Product Logs`：产品日志
   - `Products`：产品介绍
4. 新建或编辑内容。
5. 保存并发布。
6. Decap CMS 会把修改提交到 GitHub 的 `master` 分支。
7. GitHub Actions 自动构建并发布到 GitHub Pages。

发布后通常等待几十秒到几分钟，刷新站点即可看到内容。

在当前阶段，如果登录或保存失败，先回到第 3 节的本地写作发布流程。

## 5. 在线后台首次配置

GitHub Pages 是静态站点，不能自己处理 GitHub 登录回调。因此 Decap CMS 需要一个 OAuth 代理服务。

本仓库已经准备了 Cloudflare Worker 脚手架：

```text
tools/decap-oauth-worker
```

首次配置步骤见：

```text
docs/decap-cms-admin.md
```

核心配置如下：

- GitHub OAuth App 首页地址：`https://isunky.github.io`
- GitHub OAuth App 回调地址：`https://isunky-decap-oauth.isunky.workers.dev/callback`
- Decap CMS 后台配置文件：`public/admin/config.yml`
- OAuth Worker 地址：`https://isunky-decap-oauth.isunky.workers.dev`

如果后台能打开但 GitHub 登录失败，优先检查 OAuth App 回调地址和 Worker 密钥。

## 6. 写个人日志

个人日志用于记录个人思考、方法、复盘，不绑定具体产品。

本地文件目录：

```text
src/content/logs
```

Frontmatter 示例：

```md
---
title: "一篇个人日志标题"
description: "一句话概括这篇日志。"
date: 2026-06-12
tags:
  - 思考
  - 写作
draft: false
---

这里写正文。
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `title` | 是 | 文章标题 |
| `description` | 是 | 摘要，会用于列表、搜索和 SEO |
| `date` | 是 | 发布日期，格式建议用 `YYYY-MM-DD` |
| `tags` | 否 | 标签列表 |
| `draft` | 否 | 是否草稿，`true` 不公开展示 |

## 7. 写产品日志

产品日志用于记录某个产品的具体进展。首页右侧的“产品动态”和产品卡片里的最新动态都来自这里。

本地文件目录：

```text
src/content/product-logs
```

Frontmatter 示例：

```md
---
title: "MDView 增加一个新能力"
description: "记录这次迭代解决的问题和下一步计划。"
date: 2026-06-12
product: "mdview"
stage: "v0.2 规划"
version: "v0.2"
tags:
  - 产品
  - Markdown
draft: false
---

这里写产品日志正文。
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `title` | 是 | 日志标题 |
| `description` | 是 | 一句话摘要 |
| `date` | 是 | 发布日期 |
| `product` | 是 | 关联产品的 slug |
| `stage` | 是 | 阶段或版本，例如 `v0.1 规划`、`首版发布` |
| `version` | 否 | 明确版本号 |
| `tags` | 否 | 标签列表 |
| `draft` | 否 | 是否草稿 |

当前可用产品 slug：

| 产品 | slug |
| --- | --- |
| MDView | `mdview` |
| agent-md-wizard | `agent-md-wizard` |
| isunky.github.io | `isunky-github-io` |

写产品日志时，`product` 必须和产品 slug 一致，否则产品详情页无法正确关联动态。

## 8. 维护产品介绍

产品介绍用于控制 `/products/` 页面和首页“产品观察台”的产品卡片。

本地文件目录：

```text
src/content/products
```

Frontmatter 示例：

```md
---
name: "MDView"
description: "快速查看 Markdown 文件的工具。"
status: "规划中"
repo: "https://github.com/isunky/MDView"
site: "https://example.com"
stack:
  - TypeScript
  - Markdown
displayOrder: 1
draft: false
---

这里写产品的详细介绍。
```

字段说明：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `name` | 是 | 产品名称 |
| `description` | 是 | 产品短介绍 |
| `status` | 是 | 当前状态，例如 `规划中`、`迭代中`、`重建中` |
| `repo` | 是 | GitHub 仓库地址 |
| `site` | 否 | 产品站点地址 |
| `stack` | 否 | 技术栈或关键词 |
| `displayOrder` | 否 | 首页排序，数字越小越靠前 |
| `draft` | 否 | 是否草稿 |

首页默认展示排序靠前的产品。首版重点产品是：

- `MDView`
- `agent-md-wizard`
- `isunky.github.io`

每次修改站点结构、样式、脚本或内容后，建议运行：

```powershell

```

含义：

| 命令 | 作用 |
| --- | --- |

## 12. 草稿规则

三类内容都支持 `draft` 字段。

```yaml
draft: true
```

设置为 `true` 后：

- 不出现在首页。
- 不出现在列表页。
- 不进入搜索索引。
- 不进入归档和标签聚合。

准备发布时改为：

```yaml
draft: false
```

## 13. 文件命名建议

个人日志建议：

```text
YYYY-MM-DD-topic.md
```

示例：

```text
2026-06-12-writing-system.md
```

产品日志建议：

```text
product-YYYY-MM-DD-topic.md
```

示例：

```text
mdview-2026-06-12-render-plan.md
```

产品介绍文件名必须稳定，因为文件名会变成产品 slug：

```text
mdview.md
agent-md-wizard.md
isunky-github-io.md
```

不要随意改产品介绍文件名。需要改名时，同时检查所有产品日志里的 `product` 字段。

## 14. 常见问题

### 现在应该在哪里写日志

当前先在本地 Markdown 文件里写：

- 个人日志：`src/content/logs`
- 产品日志：`src/content/product-logs`
- 产品介绍：`src/content/products`

写完后按第 3 节流程提交发布。

### 后台能打开，但 GitHub 登录失败

这是当前已知状态。在线后台页面已经部署，但 GitHub 登录发布依赖 Cloudflare Worker OAuth 代理。代理服务配置完成前，请先使用本地 Markdown 写作方式。

检查：

- Cloudflare Worker 是否已部署。
- GitHub OAuth App 的 callback URL 是否正确。
- Worker 是否配置了 `GITHUB_CLIENT_ID` 和 `GITHUB_CLIENT_SECRET`。
- `public/admin/config.yml` 里的 `base_url` 是否是当前 Worker 地址。

### 新文章发布后首页没显示

检查：

- `draft` 是否为 `false`。
- `date` 是否填写正确。
- 产品日志的 `product` 是否匹配产品 slug。
- GitHub Actions 是否部署成功。

### 产品详情页没有关联日志

检查产品日志里的：

```yaml
product: "mdview"
```

这个值必须等于 `src/content/products` 里对应产品的文件名，不包含 `.md`。

### 标签页或搜索页没有出现新内容

检查：

- 是否是草稿。
- 是否已经完成部署。
- `description` 和 `tags` 是否填写。

### GitHub Actions 里有一个 Jekyll Pages 任务失败

当前站点使用自定义 Astro workflow 发布。重点看名为 `Deploy GitHub Pages` 的 workflow。

如果 `Deploy GitHub Pages` 成功，站点发布就是成功的。

## 15. 推荐写作习惯

- 个人日志写“思考和复盘”。
- 产品日志写“产品变化、问题、决策和下一步”。
- 产品介绍保持稳定，不要把每次更新都写进产品介绍。
- 产品更新优先写产品日志，让首页动态自动生成。
- 摘要控制在一两句话，方便首页、搜索和列表页展示。
- 标签不要太碎，优先复用已有标签。
